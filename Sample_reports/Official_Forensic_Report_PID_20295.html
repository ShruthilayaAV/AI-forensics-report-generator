2. EXECUTIVE SUMMARY

The investigation into Process ID 20295 centers on a targeted exploitation event directed at an internal web application identified as DVWA. Initial telemetry indicates the incident originated from a local network loopback address, suggesting either an authorized internal actor, a compromised service account, or a sophisticated attacker pivoting through a local machine. The incident commenced on June 1, 2026, at precisely 12:26:47 IST.

The attack methodology utilized a series of reconnaissance requests followed by a deliberate exploitation of a File Inclusion vulnerability. The threat actor successfully manipulated the page parameter within the vulnerable application structure, ultimately escalating the scope of the request to read sensitive system configuration files. The blast radius of this compromise is contained within the application server context, but the successful retrieval of the etc/passwd file indicates a significant failure in input sanitization and directory traversal protection.

While PID 20295 did not initiate a persistent backdoor or remote command execution shell in this specific sequence, the potential for further lateral movement is high. The business impact of this incident is classified as moderate, primarily due to the exposure of local system account structures, which provides the adversary with critical information necessary for local privilege escalation. The actor utilized a standard Mozilla browser footprint, which, while appearing innocuous, masked the delivery of malformed path traversal strings. Our forensic analysis confirms the integrity of the system was challenged, as the attacker moved from standard login procedures to a clear path traversal objective. Immediate remediation steps were necessitated to ensure that no secondary processes were spawned as a result of this interaction. The following sections detail the granular telemetry and investigative steps taken to neutralize the risk associated with this specific event, providing a roadmap for future mitigation and institutional security hardening.

6. CORE FORENSIC FINDINGS

The forensic analysis confirms that the system was subjected to a Directory Traversal attack leveraging the File Inclusion (FI) vulnerability in the target web application. The core finding is that the input parameter page was not sanitized by the application layer, allowing the attacker to escape the web directory and access the root filesystem.

The primary indicators of compromise identified are as follows:

First, a series of reconnaissance requests were performed to identify the path structure. The attacker first attempted an inclusion request for include.php, receiving a 302 redirect, which confirmed the existence of the parameter. Second, the actor successfully authenticated via the login.php page, establishing a valid session. This is a critical forensic observation because it indicates the attacker likely gained access to the application’s administrative or restricted session environment before launching the traversal attack, thereby bypassing generic unauthenticated request filters.

Third, the final payload, consisting of the string /../../../../../../etc/passwd, was successfully executed. The web server responded with a 200 OK code, meaning the application successfully interpreted the file and rendered its content back to the requester. This confirms an unauthorized disclosure of system identity information. The payload is distinct and clearly malicious, matching known patterns for Path Traversal exploitation.

Furthermore, we found no evidence of brute force on the login page prior to the 302 redirect, which indicates the attacker may have already possessed valid credentials or exploited a session fixation vulnerability. The timing between the login success and the file access (from 12:26:55 to 12:27:28) suggests a manual intervention rather than an automated botnet script, as the interval between actions aligns with human interaction speeds. There is no evidence that the /etc/passwd file content was exfiltrated to an external remote IP, as the request originated from the loopback interface (127.0.0.1). This suggests the attacker was operating locally, either through a remote desktop connection or via a previously established proxy tunnel on the machine. This finding changes the incident status from a remote network breach to a local account compromise or an internal threat scenario, necessitating a review of all physical and remote access logs for the machine hosting PID 20295.

8. DETAILED TECHNICAL EVIDENCE SUPPORTING

The technical validation of the intrusion relies on the interpretation of the HTTP request headers and the associated query strings captured in the server logs. The string /../../../../../../etc/passwd is a classic demonstration of Directory Traversal, where the attacker uses a sequence of dot-dot-slash characters to navigate upward from the web root directory to the root filesystem. Each set of dots represents a step upward in the directory hierarchy. By chaining these, the attacker escapes the DocumentRoot of the Apache server.

When the server processes this query, the PHP interpreter (libphp8.so) takes the page parameter and performs an include function. Because the input was not sanitized to strip out path-traversal sequences, the server interpreted the full string as a system path. The fact that the server returned a 200 OK status indicates that the process had sufficient file system permissions to access the etc/passwd file. This is a major configuration error. The web service user, www-data, should never have read access to the global system password file.

Furthermore, the lack of encoding on the payload suggests the attacker was confident in the server's inability to detect path traversal attempts. The presence of the referer header, which points back to the internal index page, confirms the continuity of the session. The forensic significance of the 2100-byte payload response is that it contains the user account mapping of the server, which is the exact metadata an attacker needs to begin targeted password cracking or further privilege escalation. The raw data logs were analyzed using regular expression pattern matching to identify these specific sequences, and the alignment between the session cookie and the traversal request provides absolute evidence of malicious intent from an authenticated user context.

9. ATTACKER METHODOLOGY AND ATTACK TAXONOMY

The adversary's methodology conforms to the MITRE ATT&CK framework, specifically mapping to T1059 (Command and Scripting Interpreter), T1210 (Exploitation of Remote Services), and T1592 (Gather Victim Host Information). The attack taxonomy is categorized as a Web Application Injection vulnerability, specifically Path Traversal/Local File Inclusion (LFI).

The adversary demonstrated a systematic, multi-stage approach. Initially, they performed reconnaissance (Stage 1) to identify the server's vulnerability surface. By attempting to call an inclusion file, they validated that the server was susceptible to direct script inclusion. The second phase involved credential acquisition (Stage 2) via the login page. Whether the actor utilized social engineering, brute force, or credential stuffing, the result was a valid session that provided the necessary access to the target directory. The final stage was exploitation (Stage 3), where the attacker moved from passive reconnaissance to active data retrieval, specifically targeting sensitive system files.

The attacker's objectives appear to be information gathering for future system exploitation. By retrieving the etc/passwd file, they obtained user account listings, shell accessibility status, and home directory structures. This is a classic precursor to a more extensive attack, potentially involving the deployment of a web shell if the attacker could write to the server's directory, or a privilege escalation attempt if a local vulnerability exists in one of the identified users. The use of a standard, non-obfuscated request string suggests a low-to-medium skill level attacker who is leveraging well-known vulnerability patterns rather than novel, zero-day exploits.

10. USER APPLICATIONS & SOFTWARE SUBSYSTEMS AFFECTED

The software stack affected by PID 20295 is primarily the DVWA application, which operates within an Apache2 web server environment on an Linux-based operating system. The application relies on PHP 8.x for its backend script processing and utilizes a local MySQL database, although the database was not the primary target of this specific intrusion.

The vulnerability stems from the interaction between the PHP include function and the Apache directory structure configuration. The web server was configured to allow the www-data service user excessive read permissions across the host operating system, which is a major subsystem vulnerability. The Apache configuration failed to restrict the scope of the file inclusion, effectively leaving the root directory exposed to the web-server execution context. No secondary software components such as Node.js, Python, or Ruby were involved in this specific execution path, though these languages are often targeted in similar environments. The application's own internal security mechanisms were entirely bypassed because the attacker operated within the validated session, which essentially gave them a free pass to access the vulnerable PHP module without triggering any further web application firewall (WAF) or intrusion detection alerts.

13. REMEDIATION AND IMMEDIATE INCIDENT CONTAINMENT

The immediate containment strategy for the threat presented by PID 20295 requires a multi-tiered approach. First, the Apache service must be placed into a maintenance state, and the PID 20295 process must be terminated using the SIGKILL signal to prevent any further interaction with the file system.

Second, the web server configuration must be updated to implement chroot jail isolation, which will restrict the web service process to its own directory and prevent it from navigating to sensitive system files like /etc/passwd. This is the most effective way to eliminate the root cause of the path traversal.

Third, the session identifiers generated during the incident window must be invalidated immediately. All users who were active during the 12:26 to 12:30 IST window must have their credentials reset as a precautionary measure.

Fourth, the firewall policy must be updated to restrict access to the web server to authorized IP ranges only, blocking all non-essential traffic. Lastly, the web server's file permissions should be audited and hardened to ensure that the www-data user has the minimum necessary privileges for operation, specifically removing read access to any sensitive system configuration files. These actions, if taken within the hour, will effectively neutralize the immediate threat and secure the system against similar future exploits.

14. LONG-TERM SECURITY RECOMMENDATIONS

The long-term security road map must focus on the principles of Zero Trust and Defense in Depth. The current incident demonstrates a catastrophic failure in input sanitization and privilege management. Therefore, the first strategic recommendation is to implement a robust, automated Input Validation framework for all web applications. Developers must be prohibited from using raw user-provided input in any server-side file operations.

Second, we must migrate the web server to a containerized environment (e.g., Docker or Kubernetes). Containerization provides native, hardware-level isolation that prevents a single compromised process from accessing the host's critical filesystem or other service containers, effectively mitigating the blast radius of any future path traversal exploits.

Third, we recommend the deployment of a modern Web Application Firewall (WAF) capable of deep packet inspection. A WAF would have identified the sequence of dot-dot-slash characters as a known attack pattern and blocked the request in real-time, preventing the exposure of the system files.

Fourth, implement a continuous monitoring and logging solution (SIEM) that utilizes anomaly detection based on baseline behavior. The jump from standard login to a directory traversal request is a high-confidence indicator of malicious behavior that can be alerted on in real-time.

Finally, we advocate for the adoption of the Principle of Least Privilege (PoLP) across the entire infrastructure. Regular audits of file system permissions must be performed to ensure that service accounts are strictly limited to the directories they require for operation. By implementing these changes, the organization will drastically reduce the attack surface and ensure that future vulnerabilities, even if identified, cannot be exploited to compromise the integrity of the host environment. This comprehensive strategy will transform our current reactive posture into a proactive, hardened security framework.
